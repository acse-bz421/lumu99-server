# Lumu99 Forum Backend Design

**Date:** 2026-03-11  
**Scope:** Backend API + complete API documentation (Swagger/OpenAPI + business docs)

## 1. Goals and Constraints

### 1.1 Goals
- Build backend for fan community site with admin governance.
- Deliver complete backend APIs (no admin frontend in this phase).
- Deliver full API documentation:
  - Swagger/OpenAPI
  - Business docs: permission matrix, state transitions, error codes, request/response examples.

### 1.2 Constraints
- Tech stack: Java 17 + Spring Boot + MySQL.
- API paths must **not** use `/api` prefix.
- V1 does **not** use Redis.
- Detailed runtime logs are required.
- Audit logs are required for:
  - Admin operations
  - User-sensitive operations (username/password change, account deactivation)
- Password-related fields must be masked in all logs/audits.

## 2. Product Scope

### 2.1 Content Modules
Modules:
- Story Line
- Timeline
- Photo Collection
- Video Collection
- World
- Events

Rules:
- Publish/Edit/Delete: admin only.
- Guest visibility:
  - Always visible: Story Line, Timeline, Photo Collection, Video Collection
  - Admin-configurable: World, Events
- Admin can pin/unpin content where applicable.

### 2.2 Forum Module
- Forum is login-only visible.
- Features:
  - Post creation
  - Comment
  - Vote (like/dislike, mutually exclusive, cancelable)
  - Tag selection/filtering
  - Private message
- Tag governance:
  - Admin manages tags
  - Each tag can be marked as:
    - available to all users
    - admin-only

### 2.3 Private Message Rules
- User <-> Admin DM: always allowed.
- User <-> User DM: controlled by admin switch.

## 3. Identity, Auth, and Account Governance

### 3.1 Registration
Two registration paths:
1. Quiz registration (required by default)
- Admin configures:
  - question bank
  - number of random questions per registration
- Question types:
  - single choice
  - fill-in-the-blank
- Fill-in-the-blank answer check: **exact match**.

2. Invite-code registration
- Invite code generated by admin.
- Valid invite code allows registration without quiz.

Required registration fields:
- username (unique)
- password (salted hash storage)
- weibo_name (unique)

### 3.2 User Identity Model
- Each account has backend UUID (`user_uuid`) as stable identity key.
- Internal numeric ID can still exist for DB performance.

### 3.3 Account Operations
User self-service:
- Change username
- Change password
- Deactivate account (logical delete, no physical delete)

Admin operations:
- Ban/Unban account
- Mute/Unmute account
- Force reset username
- Force reset password
- Pin/Unpin content/posts

Status rules:
- Ban: user cannot login.
- Mute: only forum post/comment disabled; DM and votes remain available.

## 4. Review Center Design

Review Center is an internal backend module (not separate service) for post moderation workflow.

### 4.1 Review Target
- Forum posts only.
- Comments are never reviewed.

### 4.2 Review Triggers
A post enters review queue when:
- it hits forbidden words (mandatory), or
- admin has enabled "user posts require review" and post is by normal user.

Admin post default behavior:
- direct publish.

### 4.3 Review Actions
- Approve
- Reject

### 4.4 Post Visibility by Review Status
- Only `APPROVED` posts are publicly visible in forum list.
- Author can view own non-approved posts in personal space (optional but recommended).

## 5. Role and Permission Matrix

Roles:
- Guest
- User
- Admin

High-level rules:
- Guest:
  - read-only access to guest-visible modules
  - no forum access
- User:
  - forum read/write (subject to mute/review rules)
  - DM and voting rights
- Admin:
  - full management rights for content, governance, and moderation

## 6. Architecture (Modular Monolith + Review Center)

Package-level modules inside one Spring Boot service:
- `auth-user`: login/register/JWT/account status
- `admin-config`: switches, forbidden words, quiz config, invite codes, tags
- `content`: 6 content modules
- `forum`: posts/comments/tags/pin
- `review-center`: queue and moderation actions
- `message`: DM threads/messages
- `interaction`: voting for forum/photo/video/world
- `audit-observability`: audit trail + logging standards

Layering:
- controller -> service -> repository -> DB

## 8. API Groups (No `/api` Prefix)

### 8.1 Auth/User
- `POST /auth/register/quiz/start`
- `POST /auth/register/quiz/submit`
- `POST /auth/register/invite`
- `POST /auth/login`
- `POST /auth/logout`
- `PUT /users/me/username`
- `PUT /users/me/password`
- `DELETE /users/me`

### 8.2 Admin User Governance
- `PUT /admin/users/{userUuid}/ban`
- `PUT /admin/users/{userUuid}/mute`
- `PUT /admin/users/{userUuid}/reset-username`
- `PUT /admin/users/{userUuid}/reset-password`

### 8.3 Admin Global Settings
- `GET /admin/settings`
- `PUT /admin/settings`

### 8.4 Admin Resources
- Forbidden words CRUD: `/admin/forbidden-words`
- Forum tags CRUD: `/admin/forum-tags`
- Quiz questions CRUD: `/admin/quiz/questions`
- Quiz config update: `PUT /admin/quiz/config`
- Invite code create/list: `/admin/invite-codes`

### 8.5 Forum
- `GET /forum/posts`
- `POST /forum/posts`
- `GET /forum/posts/{postId}`
- `PUT /forum/posts/{postId}`
- `DELETE /forum/posts/{postId}`
- `GET /forum/posts/{postId}/comments`
- `POST /forum/posts/{postId}/comments`
- `POST /forum/posts/{postId}/vote`
- `POST /forum/posts/{postId}/pin`
- `DELETE /forum/posts/{postId}/pin`

### 8.6 Review Center
- `GET /admin/reviews/posts`
- `GET /admin/reviews/posts/{postId}`
- `POST /admin/reviews/posts/{postId}/approve`
- `POST /admin/reviews/posts/{postId}/reject`

### 8.7 Content
- Stories: `/stories`
- Timelines: `/timelines`
- Photos: `/photos`
- Videos: `/videos`
- Worlds: `/worlds`
- Events: `/events`

Each supports:
- read APIs (by visibility rules)
- admin write APIs (create/update/delete/pin)

### 8.8 Non-forum Voting
- `POST /photos/{id}/vote`
- `POST /videos/{id}/vote`
- `POST /worlds/{id}/vote`

### 8.9 Messaging
- `GET /messages/threads`
- `GET /messages/threads/{threadId}`
- `POST /messages/to/{userUuid}`

## 9. State Transitions

### 9.1 Account
- `ACTIVE -> BANNED -> ACTIVE`
- `ACTIVE|BANNED -> DEACTIVATED`

### 9.2 Mute
- `NORMAL <-> MUTED`

### 9.3 Review
- `PENDING -> APPROVED|REJECTED`

### 9.4 Invite Codes
- `UNUSED -> USED|EXPIRED|DISABLED`

### 9.5 Vote
- `NONE -> LIKE|DISLIKE`
- `LIKE -> NONE|DISLIKE`
- `DISLIKE -> NONE|LIKE`

## 10. Error Code Standard

Response shape:
- `{ "code": "...", "message": "...", "requestId": "...", "data": ... }`

Examples:
- `AUTH_401_UNAUTHORIZED`
- `AUTH_403_FORBIDDEN`
- `REG_409_USERNAME_EXISTS`
- `REG_409_WEIBO_EXISTS`
- `REG_422_QUIZ_ANSWER_WRONG`
- `REG_422_INVITE_CODE_INVALID`
- `LOGIN_401_INVALID_CREDENTIALS`
- `LOGIN_403_ACCOUNT_BANNED`
- `LOGIN_403_ACCOUNT_DEACTIVATED`
- `FORUM_403_MUTED`
- `FORUM_422_TAG_NOT_ALLOWED`
- `REVIEW_404_POST_NOT_PENDING`
- `REVIEW_422_FORBIDDEN_WORD_HIT`
- `MSG_403_USER_DM_DISABLED`
- `ADMIN_403_ONLY_ADMIN`

## 11. Logging and Audit Policy

### 11.1 Runtime Logs
- Must include: requestId, method, path, operatorUuid, role, status, latencyMs.
- Log levels:
  - INFO: major business path
  - WARN: recoverable or blocked actions
  - ERROR: unexpected failures
- Password fields must always be masked.

### 11.2 Audit Logs
- Scope:
  - all admin operations
  - user-sensitive operations (change username/password, deactivate account)
- Stored in `audit_logs` table.
- Payload is plaintext except password fields masked.

## 12. Test and Acceptance

### 12.1 Test Layers
- Unit tests
- Integration tests
- Contract tests (OpenAPI consistency)
- Security tests (permission and status enforcement)

### 12.2 Mandatory Cases
- Quiz registration and invite registration
- Unique constraints (username, weibo_name)
- Ban/mute behavior
- Review center triggers and decisions
- Vote mutual exclusion + cancellation
- DM control switches
- Password masking in logs and audit logs

### 12.3 Definition of Done
- Backend APIs complete per scope.
- Swagger/OpenAPI available and accurate.
- Business docs complete (permissions, transitions, errors, examples).
- All mandatory tests pass.
